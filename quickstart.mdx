This guide will help you explore the iOS Clean Architecture example project and understand its structure.

## Prerequisites

Before you begin, ensure you have:

- **Xcode 16+** installed (required for Swift 6.2)
- **macOS** with iOS 18+ SDK
- Basic understanding of **SwiftUI** and **iOS development**

## Getting started

<Steps>
  <Step title="Clone the repository">
    Clone the project from GitHub:
    
    ```bash
    git clone https://github.com/joshgallantt/Real-Clean-Architecture-in-iOS-Example.git
    cd Real-Clean-Architecture-in-iOS-Example
    ```
  </Step>
  
  <Step title="Open in Xcode">
    Open the Xcode project:
    
    ```bash
    open CleanArchitecture.xcodeproj
    ```
    
    Or double-click `CleanArchitecture.xcodeproj` in Finder.
  </Step>
  
  <Step title="Build and run">
    1. Select the **iPhone** scheme from the scheme selector
    2. Choose a simulator (iPhone 16 Pro recommended)
    3. Press **⌘ + R** to build and run
    
    The app will launch showing a login screen.
  </Step>
  
  <Step title="Explore the app">
    The example app demonstrates a simple e-commerce flow:
    
    - **Login screen** - Enter any username/password (uses fake auth)
    - **Home tab** - Browse items with detail navigation
    - **Wishlist tab** - View wishlist items
    - **Cart tab** - Manage shopping cart
    
    Each feature demonstrates Clean Architecture principles in action.
  </Step>
</Steps>

## Project structure

The project is organized into Swift Package Manager modules, each with a specific responsibility:

<CodeGroup>
```swift User Module
User/
├── Package.swift
├── Sources/
│   ├── Domain/           # Pure business logic
│   │   ├── Model/        # Entities (User, LoginError)
│   │   ├── UseCases/     # Use cases
│   │   └── Repository/   # Repository contracts
│   ├── Data/             # Data layer
│   │   ├── Auth/         # AuthClient protocol
│   │   ├── Session/      # UserSession
│   │   └── DefaultUserRepository.swift
│   └── DI/
│       └── UserDI.swift  # Dependency injection
└── Tests/
```

```swift Feature UI Module
LoginUI/
├── Package.swift
├── Sources/
│   ├── UI/
│   │   └── LoginScreen/
│   │       ├── LoginScreenView.swift
│   │       └── LoginScreenViewModel.swift
│   └── DI/
│       └── LoginUIDI.swift
└── Tests/
```

```swift Application Layer
iPhone/
├── Injector.swift        # Composition root
├── Main/
│   ├── Main.swift        # App entry point
│   └── TabScreen.swift
└── Navigation/
    ├── Navigator.swift   # Navigation coordinator
    └── Destination.swift # Route definitions
```
</CodeGroup>

## Tracing a login flow

Let's trace what happens when a user logs in to understand how the layers work together:

<Steps>
  <Step title="User enters credentials">
    User types username/password in `LoginScreenView` and taps the login button:
    
    ```swift LoginUI/Sources/UI/LoginScreen/LoginScreenView.swift
    Button("Login") {
        Task {
            await viewModel.login()
        }
    }
    ```
  </Step>
  
  <Step title="ViewModel processes action">
    `LoginScreenViewModel` handles the action and calls the use case:
    
    ```swift LoginUI/Sources/UI/LoginScreen/LoginScreenViewModel.swift
    func login() async {
        isLoading = true
        let result = await userLogin.execute(
            username: username, 
            password: password
        )
        // Handle result...
    }
    ```
  </Step>
  
  <Step title="Use case validates input">
    `UserLoginUseCase` validates the input and delegates to the repository:
    
    ```swift User/Sources/Domain/UseCases/UserLoginUseCase.swift
    public func execute(username: String, password: String) async -> Result<Void, LoginError> {
        if username.isEmpty {
            return .failure(.usernameIsEmpty)
        }
        if password.isEmpty {
            return .failure(.passwordIsEmpty)
        }
        return await userRepository.login(username: username, password: password)
    }
    ```
  </Step>
  
  <Step title="Repository coordinates data">
    `DefaultUserRepository` calls the auth client and manages the session:
    
    ```swift User/Sources/Data/DefaultUserRepository.swift
    public func login(username: String, password: String) async -> Result<Void, LoginError> {
        let result = await authClient.login(username: username, password: password)
        switch result {
        case .success(let (user, token)):
            session.setUser(user, token: token)
            return .success(())
        case .failure(let error):
            return .failure(error.toLoginError())
        }
    }
    ```
  </Step>
  
  <Step title="Data source performs authentication">
    `FakeAuthClient` simulates authentication (in a real app, this would call an API):
    
    ```swift User/Sources/Data/Auth/FakeAuthClient.swift
    actor FakeAuthClient: AuthClient {
        func login(username: String, password: String) async -> Result<(User, AuthToken), AuthClientError> {
            // Simulate network delay
            try? await Task.sleep(for: .seconds(1))
            
            // Create fake user and token
            let user = User(id: UUID(), username: username)
            let token = AuthToken(token: "fake-jwt-token", expiresAt: Date().addingTimeInterval(3600))
            
            return .success((user, token))
        }
    }
    ```
  </Step>
</Steps>

<Note>
  Notice how each layer has a single responsibility and dependencies flow inward. The domain layer (use case) doesn't know about SwiftUI or network calls.
</Note>

## Exploring the codebase

<Accordion title="Start with the domain layer">
  Open `User/Sources/Domain/` to see the core business logic:
  
  - **Entities**: `User.swift`, `LoginError.swift`
  - **Use cases**: `UserLoginUseCase.swift`, `UserIsLoggedInUseCase.swift`
  - **Contracts**: `UserRepository.swift`
  
  These files have zero dependencies on frameworks or UI code.
</Accordion>

<Accordion title="Examine the data layer">
  Open `User/Sources/Data/` to see how data is accessed:
  
  - **Repository**: `DefaultUserRepository.swift` implements the domain contract
  - **Data sources**: `AuthClient.swift`, `UserSession.swift`
  - **DTOs**: `AuthToken.swift`
</Accordion>

<Accordion title="Study the presentation layer">
  Open `LoginUI/Sources/UI/LoginScreen/` to see the MVVM pattern:
  
  - **ViewModel**: `LoginScreenViewModel.swift` manages state and coordinates with use cases
  - **View**: `LoginScreenView.swift` is a pure SwiftUI view
</Accordion>

<Accordion title="Understand dependency injection">
  Open the application layer to see how everything is wired:
  
  - **Injector**: `iPhone/Injector.swift` - composition root that creates all dependencies
  - **UserDI**: `User/Sources/DI/UserDI.swift` - creates domain-level dependencies
  - **LoginUIDI**: `LoginUI/Sources/DI/LoginUIDI.swift` - creates UI-level dependencies
</Accordion>
